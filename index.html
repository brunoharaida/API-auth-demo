API Authentication — Interactive Demo

Scenario

You're calling a weather API (like OpenWeatherMap). You have an API key from your account dashboard.

Request

1Choose endpoint
2Header is set automatically
GET /api/weather?city=Dallas
Host: api.openweathermap.org
x-api-key: sk_live_a7f3k9m2x
3Send the request

What just happened

The server read the x-api-key header, looked it up in its database, verified it was valid, and returned data. The key never appeared in the URL — so it's not in server logs.

Scenario

A user logs into your app. After login, you get a JWT to use on protected routes.

Step 1 — Login

POST /auth/login
Body: {
  "email": "student@university.edu",
  "password": "••••••••"
}

Step 2 — Use the token on a protected route

Token stored in memory. Now attach it to your next request:

GET /api/profile
Authorization: Bearer eyJhbGci...

Key difference from API key

The token encodes the user's identity and expires — no database lookup needed. The server just verifies the signature. When it expires (e.g. 1 hour), the user must log in again.

Scenario

A developer forgets to include auth headers. This is the most common mistake.

Broken request
